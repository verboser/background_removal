# Заметки по реализации и исследованию

Задача: принять одно изображение и вернуть PNG с прозрачным фоном. Я не обучал модель с нуля. Основная работа здесь: выбрать готовую модель, быстро собрать сервис и проверить качество хотя бы на небольшом, но воспроизводимом наборе с масками.

## Короткий обзор моделей

Слово SOTA для этой задачи немного скользкое. В статьях обычно сравнивают не ровно “удали фон из любой пользовательской фотки”, а соседние задачи:

- salient object segmentation: найти главный объект и отделить фон;
- dichotomous image segmentation: аккуратно отделить foreground/background для объектов с тонкими деталями;
- image matting: получить качественную альфу на волосах, шерсти, прозрачных и полупрозрачных областях.

Я не воспроизводил метрики из статей и не пытался доказать новый SOTA. За два-три дня это было бы нечестно. Я сделал более прикладную вещь: быстро отобрал несколько кандидатов, оставил два рабочих режима и проверил их на одном фиксированном протоколе.

| Кандидат | Почему смотрел | Что остановило или ограничило | Что сделал |
| --- | --- | --- | --- |
| U2-Net | классический baseline для saliency | старый подход, на шерсти и тонких краях обычно грубее | не стал тратить время |
| MODNet | хорош для портретов и волос | слишком портретная модель, не универсальная для животных и товаров | отложил |
| SAM / SAM 2 | сильная интерактивная сегментация | нужен prompt или отдельный выбор объекта, альфы из коробки нет | не подходит для one-click сервиса |
| IS-Net / DIS-подходы | близко к задаче dichotomous segmentation | нужно отдельно поднимать веса и проверять интеграцию | оставил как следующий шаг |
| BRIA RMBG-2.0 | выглядит сильным именно для background removal | model card указывает non-commercial ограничения для весов | не использовал в коде |
| rembg + birefnet-general | быстро дает рабочий baseline | меньше контроля над препроцессингом, есть overhead | оставил baseline |
| ZhengPeng7/BiRefNet | можно запустить напрямую через PyTorch | CPU все равно медленный, прозрачные объекты не проверены | оставил основной режим для сравнения |

Практический вывод получился такой: для прототипа важнее было не “самая свежая модель из статьи”, а модель, которую можно поднять локально, прогнать на реальных масках и показать глазами, где она ошибается.

Источники, которые смотрел:

- U2-Net paper: https://arxiv.org/abs/2005.09007
- rembg repository: https://github.com/danielgatis/rembg
- BiRefNet paper: https://arxiv.org/abs/2401.03407
- BiRefNet model card: https://huggingface.co/ZhengPeng7/BiRefNet
- BRIA RMBG-2.0 model card: https://huggingface.co/briaai/RMBG-2.0
- Segment Anything repository: https://github.com/facebookresearch/segment-anything
- MODNet paper: https://arxiv.org/abs/2011.11961

## Критерии выбора

Я отбирал модель по таким критериям:

- работает без prompt-а и без ручной разметки;
- возвращает маску, которую можно сразу положить в alpha channel;
- есть готовые веса;
- можно запустить локально на CPU, пусть и медленно;
- лицензия не блокирует тестовое использование;
- ожидаемо справляется с шерстью и тонкими краями лучше старых saliency-моделей.

Поэтому в коде оставлены два режима: `rembg + birefnet-general` как baseline и прямой `ZhengPeng7/BiRefNet` через PyTorch как более контролируемый вариант.

## Как работает сервис

```text
Streamlit
  -> FastAPI
    -> проверка типа и размера файла
    -> декодирование через Pillow
    -> модель
    -> PNG с альфа-каналом
```

Модель грузится один раз при старте FastAPI. Streamlit инференс не делает: отправляет файл на бэкенд, получает результат, показывает исходник и PNG с прозрачностью.

## Датасетный протокол

Для оценки я взял Oxford-IIIT Pets, потому что там есть реальные изображения животных и trimap-маски. Это не идеальный датасет для background removal в целом: он не проверяет товары, людей, стекло, прозрачные объекты и маркетплейсные сцены. Зато он хорошо ловит проблемы с шерстью, лапами, хвостами и неоднозначным контуром.

Протокол:

- split: `annotations/test.txt`;
- eval subset: `ml/oxford_pets_eval50.txt`;
- размер: 50 изображений из test split;
- выборка: первые 37 пород по одному примеру, затем дополнительные примеры по тем же породам до 50 файлов;
- картинки: `data/oxford_pets/images_eval50`;
- маски: `data/oxford_pets/annotations/trimaps`;
- сопоставление: по имени файла без расширения;
- `--strict` падает, если нет картинки или маски.

В trimap Oxford Pets класс `2` я считаю фоном. Классы `1` и `3` считаю объектом. Это не идеально: класс `3` часто попадает на неопределенный край. Но для удаления фона лучше оставить такую область у объекта, чем жестко вырезать шерсть.

Команды:

```bash
python ml/evaluate.py --images data/oxford_pets/images_eval50 --masks data/oxford_pets/annotations/trimaps --file-list ml/oxford_pets_eval50.txt --backend rembg --out outputs/rembg_oxford_pets_eval50.csv --summary outputs/rembg_oxford_pets_eval50.summary.md --gallery outputs/rembg_oxford_pets_eval50_gallery.jpg --gallery-items 5 --strict
python ml/evaluate.py --images data/oxford_pets/images_eval50 --masks data/oxford_pets/annotations/trimaps --file-list ml/oxford_pets_eval50.txt --backend birefnet --out outputs/birefnet_oxford_pets_eval50.csv --summary outputs/birefnet_oxford_pets_eval50.summary.md --gallery outputs/birefnet_oxford_pets_eval50_gallery.jpg --gallery-items 5 --strict
```

## Метрики

Скрипт считает:

- MAE по мягкой маске;
- IoU и Dice после бинаризации с порогом 0.5;
- Boundary MAE около контура объекта;
- FP rate и FN rate;
- object ratio;
- latency на каждый файл, p50 и p90.

MAE сам по себе плохо показывает ошибки на маленьких объектах. Поэтому IoU/Dice и object ratio важны отдельно. Boundary MAE нужен, потому что для background removal визуально чаще всего бросается в глаза именно край.

## Результаты на Oxford Pets eval50

| Модель | MAE | IoU | Dice | Boundary MAE | Avg latency | P50 latency | P90 latency |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| rembg + birefnet-general | 0.06047 | 0.83669 | 0.90832 | 0.47923 | 5212.1 ms | 4911.9 ms | 6269.3 ms |
| ZhengPeng7/BiRefNet | 0.06119 | 0.83560 | 0.90757 | 0.47993 | 1665.2 ms | 1670.8 ms | 1716.0 ms |

По качеству разницы почти нет. На этих 50 изображениях `rembg + birefnet-general` чуть лучше по MAE/IoU/Dice, но разрыв слишком маленький, чтобы объявлять победителя. По времени прямой BiRefNet заметно быстрее в моем CPU-прогоне: около 0.60 изображения/сек против 0.19 изображения/сек у rembg.

По размеру объекта:

| Модель | Малый объект IoU | Средний объект IoU | Крупный объект IoU |
| --- | ---: | ---: | ---: |
| rembg + birefnet-general | 0.73105 | 0.84729 | 0.91181 |
| ZhengPeng7/BiRefNet | 0.72365 | 0.84849 | 0.90838 |

Маленькие объекты проседают сильнее всего. Это важнее средней MAE: если объект занимает мало места, небольшая абсолютная ошибка может сильно испортить IoU и визуальный результат.

## Визуальный анализ

Галереи лежат здесь:

- `outputs/rembg_oxford_pets_eval50_gallery.jpg`;
- `outputs/birefnet_oxford_pets_eval50_gallery.jpg`.

В каждой строке четыре колонки: source, target, prediction, error. В error красный показывает место расхождения между предсказанием и trimap-derived target.

Что видно по худшим кейсам:

- `samoyed_191.jpg`: обе модели оставляют человека рядом с собакой. Для автоматического background removal это типичная ошибка: модель видит несколько связанных foreground-объектов и не знает, что пользователю нужна только собака.
- `american_bulldog_205.jpg`: вместе с собакой остается часть темной сумки/предмета около головы и лап. Это ошибка co-salient object, когда соседний предмет визуально связан с объектом.
- `Maine_Coon_222.jpg`: у прямого BiRefNet заметно больше лишнего фона справа. Пушистая шерсть и темный фон делают границу неоднозначной.
- `boxer_191.jpg`: проблемы идут по лапам и мячу. Модель не всегда понимает, является ли предмет у лап частью объекта.
- `Sphynx_211.jpg`: ошибка на области руки/тела человека. Для pet-dataset это ожидаемый конфликт: животное находится на руках, а target размечает только животное.

Вывод по визуальному анализу: обе модели скорее conservative по объекту и часто сохраняют связанные foreground-объекты. Для пользовательского сервиса это лучше, чем отрезать часть животного, но хуже для чистой товарной фотографии, где лишний предмет будет сразу заметен.

## Что покрыто и что не покрыто

Покрыто:

- животные, шерсть, лапы, хвосты;
- разные размеры объекта в кадре;
- локальный CPU latency;
- проверка худших кейсов глазами.

Не покрыто:

- люди и портреты как отдельный сценарий;
- товары на маркетплейсном фоне;
- прозрачные объекты, стекло, дым, волосы крупным планом;
- полный Oxford Pets test split;
- GPU latency и параллельная обработка запросов.

## Тесты

Быстрый прогон:

```text
pytest: 7 passed, 2 deselected
```

Он проверяет расчет метрик, перевод Oxford Pets trimap в foreground/background, boundary-метрику и ошибки evaluator до загрузки модели: нет папки, пустая папка, нет пар картинка + маска, нет файла из `--file-list` при `--strict`.

Интеграционный прогон:

```text
pytest -m integration: 2 passed, 7 deselected
```

Он грузит реальные модели и проверяет один файл из Oxford Pets eval50. Это не заменяет полный прогон, но быстро ловит ситуацию, когда модель перестала грузиться или начала отдавать совсем плохую маску.

## Итог

Для трехдневного прототипа я бы оставил оба режима. `rembg + birefnet-general` хорош как baseline и проще объясняется. Прямой BiRefNet быстрее в текущем CPU-прогоне и дает больше контроля над preprocessing/postprocessing. Для следующей итерации я бы добавил отдельный набор людей/товаров, ручной просмотр 20-30 худших кейсов и GPU-прогон на полном Oxford Pets test split.
